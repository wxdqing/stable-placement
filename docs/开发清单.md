# Stable Placement 开发清单

来源文档：

- `开发需求文档.md`
- `需求.md`
- `ontology.md`

目标：

```text
按最小可用闭环实现 Stable Placement 第一阶段。
先保证语义正确，再考虑 Redis / etcd / Dashboard / Rebalance 等扩展。
```

使用方式：

- 按阶段从上到下执行。
- 每完成一项，勾选对应 checkbox。
- 每个阶段都要求有测试或可验证结果。
- 不为了未来扩展提前增加复杂目录和抽象。

## 大致目录结构

第一阶段建议按模块分目录。

根包只放公共契约：类型、命令、接口、错误。

具体实现放到明确模块目录里。

```text
libs/stable-placement/
  go.mod

  identity.go
  model.go
  errors.go
  command.go

  directory.go
  node_registry.go
  cache.go
  event.go
  strategy.go

  identity_test.go
  model_test.go
  command_test.go
  integration_test.go

  memory/
    directory.go
    node_registry.go
    invalid_node_group.go
    cache.go
    event_bus.go

    directory_test.go
    node_registry_test.go
    cache_test.go
    event_bus_test.go

  strategies/
    round_robin.go
    round_robin_test.go

  adapters/
    protoactor/
      README.md

  docs/
    需求.md
    ontology.md
    开发需求文档.md
    开发清单.md
```

第一阶段不建议这样拆：

```text
libs/stable-placement/
  domain/
  application/
  repository/
  service/
  manager/
  provider/
```

原因：

- 模块目录可以提升可读性。
- 根包保留稳定接口，避免实现细节污染公共契约。
- 不使用 DDD 多层目录，避免 `repository/service/manager/provider` 这类不必要中间层。

后续扩展目录：

```text
libs/stable-placement/
  redis/
    directory.go
    event_bus.go

  etcd/
    directory.go
    event_bus.go

  adapters/
    protoactor/
      placement.go
```

目录职责：

- 根包：公共类型、命令、接口、错误。
- `memory/`：第一阶段内存实现。
- `strategies/`：分配策略实现。
- `adapters/`：运行时适配器。
- `redis/`、`etcd/`：后续基础设施实现。

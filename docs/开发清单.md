## 大致目录结构

第一阶段建议按模块分目录。

根包只放公共契约：类型、命令、接口、错误。

具体实现放到明确模块目录里。

```text
libs/stable-placement/
  go.mod

  identity.go
  model.go
  errors.go
  command.go

  directory.go
  node_registry.go
  cache.go
  event.go
  strategy.go

  identity_test.go
  model_test.go
  command_test.go
  integration_test.go

  memory/
    directory.go
    node_registry.go
    invalid_node_group.go
    cache.go
    event_bus.go

    directory_test.go
    node_registry_test.go
    cache_test.go
    event_bus_test.go

  strategies/
    round_robin.go
    round_robin_test.go

  adapters/
    protoactor/
      README.md

  docs/
    需求.md
    ontology.md
    开发需求文档.md
    开发清单.md
```

第一阶段不建议这样拆：

```text
libs/stable-placement/
  domain/
  application/
  repository/
  service/
  manager/
  provider/
```

原因：

- 模块目录可以提升可读性。
- 根包保留稳定接口，避免实现细节污染公共契约。
- 不使用 DDD 多层目录，避免 `repository/service/manager/provider` 这类不必要中间层。

后续扩展目录：

```text
libs/stable-placement/
  redis/
    directory.go
    event_bus.go

  etcd/
    directory.go
    event_bus.go

  adapters/
    protoactor/
      placement.go
```

目录职责：

- 根包：公共类型、命令、接口、错误。
- `memory/`：第一阶段内存实现。
- `strategies/`：分配策略实现。
- `adapters/`：运行时适配器。
- `redis/`、`etcd/`：后续基础设施实现。

## 14. 第二阶段开发清单

第二阶段目标：

```text
把第一阶段 MemoryDirectory 的语义迁移到可共享、可恢复的外部存储。
Grain 一旦分配到节点，Directory 必须记住 Grain -> NodeIdentity 的稳定关系。
后续 Lookup 只能读取这个固化关系，不能因为节点列表或策略变化而自动重分配。
```

Dashboard 相关放到最后。

第二阶段优先级：

```text
1. 生命周期补齐：NodeDraining / Release / Expire
2. RedisDirectory
3. Redis Lua 原子写入
4. Redis Stream / outbox EventBus
5. 显式迁移与负载调整工具
6. etcd Directory / watch
7. protoactor-go adapter
8. Dashboard
```

---

### 14.1 第二阶段边界

- [ ] 保持根包接口不随 Redis / etcd 实现大幅变化。
- [ ] 保持 `Directory` 仍然是 Source of Truth。
- [ ] 保持 `PlacementStrategy` 只参与首次 Allocate。
- [ ] 保持 Lookup 无副作用。
- [ ] 保持已分配 Placement 不因节点增减自动迁移。
- [ ] 保持迁移只能通过显式 `Transfer / Release / Recover`。
- [ ] 不做自动抖动式 Rebalance。
- [ ] 第二阶段优先完成 Redis，etcd 放到 Redis 语义稳定之后。
- [ ] Redis 写操作必须保证状态变更和事件记录的事务性。
- [ ] 不引入 Repository / Service / Manager 多层结构。

---

### 14.2 NodeDraining / Release / Expire 生命周期补齐

第二阶段需要把节点缩容和 Lease 清理的生命周期补完整。

核心边界：

```text
MarkNodeInvalid:
  调度规则，禁止新的 Allocate，按 NodeName 生效

NodeDraining:
  节点生命周期状态，表示某个 NodeIdentity 正在缩容迁移

Release:
  Grain 主动结束，Owner 主动释放 Placement

Expire:
  Lease 超时，被动清理 Placement
```

#### 14.2.1 NodeDraining 独立于 MarkNodeInvalid

- [x] `MarkNodeInvalid` 只负责把 NodeName 加入 InvalidNodeGroup。
- [x] `MarkNodeInvalid` 只影响新的 Allocate。
- [x] `MarkNodeInvalid` 不表示节点正在下线。

推荐缩容流程：


### 14.5 Redis Stream / outbox EventBus

目录：


Pub/Sub 与 Stream / outbox 对比：

| 方案 | 优点 | 缺点 | 适用场景 |

推荐：

```text

```

- [x] 禁止所有运行节点共享同一个 consumer group，否则事件会被负载均衡给单个节点。
- [x] consumer group 名称必须包含 NodeIdentity 和 NodeSessionID。

- [x] 节点正常下线时删除自己的 consumer group。
- [ ] NodeSessionID 替换后，新 session 必须使用新的 consumer group。
- [ ] 旧 session 的 consumer group 必须在确认不再需要后清理。
- [ ] pending 消息超过阈值时必须进入 degraded 或重新处理。
- [x] 支持记录消费进度。
- [x] 支持 pending 消息重试。

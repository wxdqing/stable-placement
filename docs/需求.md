# Stable Placement 需求文档

> v1 历史需求：本文保留早期 Grain Lease、heartbeat、Expire 等需求背景，不是 Node Lease v2 当前契约。当前模型以 `开发需求文档.md`、`ontology.md` 和 `node-lease-v2-cutover.md` 为准；本文仍保留的运行时兼容、缓存、失效节点组、策略、可观测性和后续存储需求继续有效，除非当前文档明确取代。

## 1. 背景

Stable Placement 用于维护 Grain 的稳定归属关系。

它不应该被设计成某个 Redis 存储结构，也不应该被设计成 protoactor-go 的内部功能，而应该是一个可以被不同运行时接入的独立组件。

核心问题不是：

```text
Redis 怎么存？
```

而是：

```text
一个 Grain 当前应该由哪个节点负责？
这个归属关系如何保持唯一、稳定、可发现、可恢复、可迁移？
```

## 2. 组件定位

Stable Placement 是一个独立组件。

它负责：

- 维护 Grain 到 Node 的归属关系。
- 保证同一个 Grain 同一时间只有一个有效 Owner。
- 为调用方提供稳定的 Lookup / Allocate / Transfer / Release / Recover 能力。
- 支持节点横向扩展。
- 支持 protoactor-go 节点，也支持非 protoactor-go 节点。

它不负责：

- 创建 Actor。
- 管理 Actor 生命周期。
- 处理业务消息。
- 替代 protoactor-go remote。
- 做实时负载均衡。
- 决定业务协议。

## 3. 核心目标

Stable Placement 的目标是：

```text
保证一个 Grain 在整个生命周期内拥有稳定、唯一、可管理的 Owner。
```

关键能力：

- Stable：正常情况下归属不随重启、GC、Actor 回收而变化。
- Unique：同一个 Grain 同一时间最多只有一个有效 Owner。
- Discoverable：任意节点都可以查询 Grain 当前 Owner。
- Recoverable：Owner 节点故障后可以恢复归属。
- Transferable：需要迁移时可以显式转移归属。
- Scalable：节点可以按类型和组横向扩展。

## 4. 节点模型

节点不是只有一个简单的 NodeID。

实际使用时，节点由三部分组成：

```text
节点类型 + 组名 + 实例名字
```

建议模型：

```text
NodeType
NodeGroup
NodeName
```

组合后形成稳定节点身份：

```text
NodeIdentity = NodeType + "/" + NodeGroup + "/" + NodeName
```

示例：

```text
game/default/game-1
game/default/game-2
game/shanghai/game-1
chat/world/chat-1
battle/pve/battle-3
gateway/public/gateway-2
```

字段含义：

- NodeType：节点类型，例如 game、chat、battle、gateway。
- NodeGroup：节点组名，用于表达分组、区域、用途、灰度批次或逻辑集群。
- NodeName：实例名字，在同一个 NodeType + NodeGroup 内唯一。

约束：

- 同一个 `NodeType + NodeGroup` 下可以有多个 NodeName。
- `NodeIdentity` 必须全局唯一。
- Placement 记录里保存的是完整 NodeIdentity。
- NodeGroup 不是机器名，也不是进程 ID，而是调度和扩展维度。

## 5. 横向扩展模型

Stable Placement 需要支持按下面的维度横向扩展：

```text
NodeType + NodeGroup
```

例如：

```text
game/default:
  game-1
  game-2
  game-3

chat/world:
  chat-1
  chat-2

battle/pve:
  battle-1
  battle-2
  battle-3
```

PlacementStrategy 在分配时，应先确定候选节点集合：

```text
NodeType + NodeGroup -> Candidate Nodes
```

然后在候选集合里选择具体节点。

示例：

```text
Player10001
  Kind: Player
  Target: game/default

Strategy:
  game/default -> [game-1, game-2, game-3]
  choose game-2

Placement:
  Player10001 -> game/default/game-2
```

容量目标：

```text
支持十万在线。
```

这里的十万在线不是单机保证，而是组件支持通过增加节点横向扩展来承载：

```text
容量 = 单节点承载能力 * 节点数量
```

Stable Placement 需要保证在节点数量增加时：

- Lookup 不退化成全量扫描。
- Allocate 不依赖单点本地内存。
- Placement 数据可以被稳定查询。
- 节点增减不会导致已有 Placement 大面积抖动。
- 只有明确 Transfer / Recover / Release 才改变已有 Placement。

### 5.1 本地内存缓存

Stable Placement 可以在节点本地保存 Placement 缓存。

原因：

- Placement 是稳定分配。
- 分配完成后，Grain 的 Owner 正常情况下不会频繁变化。
- 调用方不应该每次路由都访问 Directory。
- 已经分配过的 Grain 可以安全缓存路由结果。

缓存原则：

- 缓存不是 Source of Truth。
- Directory 仍然是唯一真相。
- 缓存只保存已经分配或查询过的 Placement。
- 缓存命中时可以直接使用。
- 缓存失效或不确定时必须回退到 Directory Lookup。

建议缓存内容：

```text
GrainKey
Kind
NodeIdentity
PlacementVersion
LeaseExpireAt
Status
UpdateTime
```

其中：

```text
GrainKey = Kind + "/" + GrainID
```

节点本地可以维护：

```text
LocalPlacementCache:
  GrainKey -> PlacementRoute
```

也可以按节点维度维护索引：

```text
NodeIdentity -> GrainKey Set
```

这样在节点缩容、节点替换、节点下线时，可以按具体节点清理相关 Grain 路由缓存。

### 5.2 缓存失效机制

缓存必须支持主动失效。

典型失效来源：

- PlacementTransferred
- PlacementReleased
- PlacementRecovered
- LeaseExpired
- NodeUnregistered
- NodeDraining
- NodeReplaced
- ManualCacheClear

失效方式：

```text
Directory / Repository
  -> Publish Event
  -> Node Subscribe Event
  -> Clear Local Cache
```

第一阶段可以使用 Redis Pub/Sub 作为事件通知机制。

后续也可以替换为：

```text
etcd watch
消息队列
自研事件总线
Placement Service stream
```

缓存失效粒度：

- 按 GrainKey 清理。
- 按 NodeIdentity 清理。
- 按 NodeType + NodeGroup 清理。

其中按 NodeIdentity 清理很重要。

例如缩容时：

```text
NodeDraining(game/default/game-2)
  -> 清理所有指向 game/default/game-2 的本地缓存
  -> 后续 Lookup 回源 Directory
```

同名节点实例重新启动时，也必须触发旧节点信息清理。

例如：

```text
旧节点: game/default/game-2 session=A
新节点: game/default/game-2 session=B
```

虽然 NodeIdentity 相同，但它们不是同一个运行实例。

因此 Node 需要额外的运行期标识：

```text
NodeSessionID
```

当同名节点以新的 NodeSessionID 注册时：

```text
NodeReplaced(game/default/game-2, oldSession=A, newSession=B)
  -> 清理旧 session 相关节点信息
  -> 清理指向该 NodeIdentity 的本地缓存
  -> 旧 Owner 续约必须失败
```

缓存更新规则：

- Allocate 成功后，调用节点应写入本地缓存。
- Lookup Directory 成功后，可以写入本地缓存。
- Transfer / Release / Recover / Expire 后，必须发布失效事件。
- 收到比本地版本新的事件时，必须清理或更新本地缓存。
- 收到无法判断版本的事件时，必须保守清理缓存。

事件订阅异常时，节点不能继续信任本地缓存。

订阅异常包括：

- 订阅连接断开。
- 订阅恢复后无法确认中间事件是否丢失。
- 事件反序列化失败。
- 事件版本或顺序无法判断。
- 事件处理连续失败。

处理要求：

- 必须清空本地 Placement 缓存，或进入降级回源模式。
- 降级回源模式下，Lookup 必须跳过本地缓存，直接查询 Directory。
- 降级回源模式下，禁止写入本地缓存。
- 订阅恢复并重新确认事件连续性之前，不允许把旧缓存当作可信路由。
- 如果无法判断是否漏事件，必须按最保守策略清理缓存。

降级恢复要求：

- 订阅重新建立后，必须先清空本地缓存。
- 清空完成后，可以重新开启缓存读写。
- 如果使用 Redis Pub/Sub 这类不可回放事件源，断线重连后不能假设中间事件可补齐，只能通过清空本地缓存恢复安全状态。

### 5.3 节点失效组

为了支持缩容，需要维护节点失效组。

节点失效组用于表示：

```text
某个 NodeType + NodeGroup 下，哪些具体 NodeName 已经不允许参与新分配。
```

建议结构：

```text
InvalidNodeGroup:
  key:   NodeType + "/" + NodeGroup
  value: NodeName Set
```

示例：

```text
key: game/default
value:
  - game-2
  - game-5
```

含义：

```text
game/default/game-2 不再参与新的 Allocate。
game/default/game-5 不再参与新的 Allocate。
```

InvalidNodeGroup 只按 `NodeName` 生效，不按 `NodeSessionID` 生效。

也就是说，失效状态跨 NodeSessionID 持续有效：

```text
MarkNodeInvalid(game/default, game-2)

旧节点: game/default/game-2 session=A
新节点: game/default/game-2 session=B

结果:
  game-2 仍然在 game/default 的失效组里
  session=B 也不能参与新的 Allocate
  只有 RestoreNode(game/default, game-2) 后才恢复
```

注意：

- 失效节点不等于节点已经死亡。
- 失效节点可能仍然在线。
- 失效节点可能仍然持有旧 Placement。
- 失效节点只是不能再承接新的 Placement。
- 已经在失效节点上的旧 Placement，需要通过 Transfer / Release / Recover 慢慢迁走或清理。

失效组主要用于缩容：

```text
1. MarkInvalid(game/default, game-2)
2. 新 Allocate 不再选择 game/default/game-2
3. 已有 Placement 继续可 Lookup
4. 后台或运维流程逐步 Transfer / Release
5. game-2 没有承载后可以下线
```

分配流程必须先过滤失效节点：

```text
NodeType + NodeGroup
  -> Candidate Nodes
  -> Remove Invalid NodeName
  -> Effective Nodes
  -> Strategy Choose
```

如果过滤后没有有效节点，Allocate 必须失败，不能回退到失效节点。

## 6. 运行时兼容

Stable Placement 需要同时支持两类节点。

### 6.1 protoactor-go 节点

protoactor-go 节点接入时：

- Stable Placement 只负责找到 Grain Owner。
- protoactor-go 负责 Actor 创建、Remote 通信和消息投递。
- Placement 的 Owner 可以映射到 protoactor-go 的节点地址或 actor runtime 节点。

典型流程：

```text
Lookup Grain
  -> 得到 NodeIdentity
  -> protoactor-go remote 定位节点
  -> 节点内部激活/获取 Actor
  -> 投递消息
```

### 6.2 非 protoactor-go 节点

非 protoactor-go 节点也可以接入。

例如：

- 普通 Go 服务。
- 网关服务。
- 战斗服务。
- 聊天服务。
- 外部进程。

这类节点只需要实现：

- 注册节点。
- 续约节点或 Placement lease。
- 查询 Placement。
- 在自己成为 Owner 时处理对应业务。

Stable Placement 不假设 Owner 一定是 Actor。

Owner 可以是：

```text
Actor
Service
Process
Worker
Shard
```

## 7. 核心实体

### 7.1 Grain

逻辑实体。

示例：

```text
Player
Guild
Mail
Rank
Scene
```

关键字段：

```text
GrainID
Kind
TargetNodeType
TargetNodeGroup
```

说明：

- Grain 不关心自己在哪台机器。
- Kind 用于区分业务类型。
- TargetNodeType + TargetNodeGroup 用于确定候选节点集合。

### 7.2 Node

运行节点。

关键字段：

```text
NodeType
NodeGroup
NodeName
NodeIdentity
NodeSessionID
Address
Weight
Load
Status
LastHeartbeatAt
```

### 7.3 Placement

Grain 到 Node 的归属关系。

关键字段：

```text
GrainID
Kind
NodeIdentity
Version
Status
CreateTime
UpdateTime
LeaseExpireAt
```

Placement 是核心 Aggregate。

### 7.4 Directory

Placement 的唯一真相。

提供：

```text
Lookup
Allocate
Release
Transfer
Recover
Renew
List
Exists
FindByNode
FindByKind
FindByGroup
```

`FindByNode` 必须按完整 `NodeIdentity` 查询 Placement。

它用于缩容迁移、节点摘流和运维排查，例如：

```text
FindByNode(game/default/game-2)
  -> Player10001
  -> Player10002
  -> Guild9001
```

缩容流程可以先通过 `FindByNode(NodeIdentity)` 找到该节点当前承载的 Placement，再逐个执行显式 `Transfer / Release / Recover`。

查询要求：

- 默认只返回当前有效的 Active Placement。
- 如需排查历史数据，可以显式指定 Status 过滤条件。
- 必须支持分页或游标，避免单个节点承载过多 Placement 时一次性返回过大结果。
- 存储实现必须维护 `NodeIdentity -> Placement` 查询能力，不能依赖全量扫描。

### 7.5 Lease

Lease 用于保护唯一 Owner。

第一阶段可以作为 Placement 的内部状态存在：

```text
LeaseOwnerNodeIdentity
LeaseOwnerNodeSessionID
LeaseVersion
LeaseExpireAt
```

不一定需要作为独立存储实体暴露。

### 7.6 PlacementStrategy

负责首次放置。

输入：

```text
GrainID
Kind
NodeType
NodeGroup
CandidateNodes
```

输出：

```text
NodeIdentity
```

Strategy 不参与 Lookup。

## 8. 生命周期

### 8.1 Register Node

节点启动后注册自己：

```text
NodeType + NodeGroup + NodeName
```

注册成功后，节点成为在线节点。

是否可以参与新的 Allocate，还必须同时满足：

- 节点状态允许承载。
- 该 NodeName 不在对应 `NodeType + NodeGroup` 的 InvalidNodeGroup 中。

因此，同名节点以新的 `NodeSessionID` 注册成功，也不会自动解除失效状态。

### 8.2 Lookup

查询 Grain 当前 Placement。

要求：

- 无副作用。
- 不创建 Placement。
- 不改变 Owner。
- 可以优先读取本地内存缓存。
- 缓存命中且版本有效时，可以直接返回。
- 缓存未命中、已失效或版本不确定时，必须回源 Directory。
- 回源 Directory 成功后，可以刷新本地缓存。
- 如果节点处于降级回源模式，必须跳过本地缓存，且不得刷新本地缓存。

### 8.3 Allocate

当 Grain 没有有效 Placement 时执行。

流程：

```text
Grain
  -> Target NodeType + NodeGroup
  -> Candidate Nodes
  -> Filter Invalid NodeName
  -> Effective Nodes
  -> Strategy
  -> Create Placement
  -> Write Local Cache if cache is healthy
```

要求：

- Candidate Nodes 来自 `NodeType + NodeGroup`。
- 分配前必须过滤节点失效组。
- Strategy 只能看到过滤后的 Effective Nodes。
- 如果 Effective Nodes 为空，Allocate 必须返回无可用节点错误。

### 8.4 Activate

Owner 节点根据 Placement 激活业务对象。

如果接入 protoactor-go，则是激活 Actor。

如果不是 protoactor-go，则由业务节点自行处理。

### 8.5 Renew

Owner 续约 Placement。

要求：

- 只能当前 Owner 续约。
- Renew 请求必须携带 `NodeIdentity`、`NodeSessionID` 和当前 `Version / LeaseVersion`。
- Directory 必须校验 `NodeSessionID` 是该 `NodeIdentity` 当前有效的运行实例。
- Version / Lease 必须匹配。
- 续约失败说明 Owner 可能已经失效或被迁移。

### 8.6 Release

Owner 主动释放 Placement。

例如：

- 玩家下线。
- 节点主动关闭。
- 业务主动释放。

要求：

- Release 必须携带 `NodeIdentity`、`NodeSessionID` 和当前 `Version / LeaseVersion`。
- Directory 必须校验请求来自当前有效 Owner session。

### 8.7 Transfer

显式迁移 Placement。

例如：

```text
game/default/game-2 -> game/default/game-5
```

Transfer 必须是显式命令。

Lookup 不能触发 Transfer。

缩容迁移时，调用方应先按 `NodeIdentity` 查询待迁移 Placement：

```text
FindByNode(game/default/game-2)
  -> Transfer each Placement to other effective nodes
```

迁移目标必须来自过滤失效节点后的 Effective Nodes。

### 8.8 Recover

节点故障后恢复 Placement。

Recover 与 Transfer 不同：

- Transfer 是有计划迁移。
- Recover 是旧 Owner 不可信时的故障恢复。

### 8.9 Cache Invalidate

缓存失效由事件驱动。

流程：

```text
Placement Changed
  -> Publish Event
  -> Related Nodes Receive Event
  -> Clear Local Cache
```

事件可以来自：

- Redis Pub/Sub。
- etcd watch。
- 消息队列。
- Placement Service stream。

清理范围可以是：

```text
GrainKey
NodeIdentity
NodeType + NodeGroup
```

缩容时，应该主动清理目标节点相关缓存：

```text
NodeDraining(NodeIdentity)
  -> Clear cache by NodeIdentity
```

同名节点新实例启动时，应该清理旧节点信息：

```text
NodeReplaced(NodeIdentity, oldSession, newSession)
  -> Clear cache by NodeIdentity
  -> Reject old session renew
```

事件订阅异常时，节点必须立即清空本地缓存或降级为只读回源 Directory。

```text
Subscribe Error / Event Gap / Decode Failed
  -> Clear Local Cache
  -> Lookup fallback to Directory
  -> Disable cache write until subscription is healthy
```

### 8.10 Mark Node Invalid

缩容或摘流时，可以将具体节点加入失效组。

示例：

```text
MarkNodeInvalid(game/default, game-2)
```

效果：

- `game/default/game-2` 不再参与新的 Allocate。
- 失效状态按 NodeName 保存，跨 NodeSessionID 持续有效。
- 已有 Placement 不会因为 MarkInvalid 立即改变。
- 后续可以通过 Transfer / Release / Recover 清理已有 Placement。
- 本地缓存可以按 NodeIdentity 主动清理，促使后续 Lookup 回源 Directory。
- 必须发布 `NodeMarkedInvalid` 事件。

事件内容至少包含：

```text
NodeType
NodeGroup
NodeName
NodeIdentity
```

### 8.11 Restore Node

如果缩容取消或节点重新允许承载，可以从失效组移除节点。

示例：

```text
RestoreNode(game/default, game-2)
```

效果：

- `game/default/game-2` 重新进入 Effective Nodes。
- 后续 Allocate 可以再次选择该节点。
- 必须发布 `NodeRestored` 事件。

事件内容至少包含：

```text
NodeType
NodeGroup
NodeName
NodeIdentity
```

## 9. 领域规则

### Rule 1: Grain 唯一归属

同一个 Grain 同一时间最多只有一个 Active Placement。

### Rule 2: Lookup 无副作用

Lookup 只能读取 Placement，不能创建、迁移或恢复 Placement。

### Rule 3: Placement 稳定

正常情况下，Placement 不因为 Actor GC、Actor 回收、普通重启而变化。

### Rule 4: Placement 只能被 Command 改变

允许改变 Placement 的命令包括：

```text
Allocate
Release
Transfer
Recover
Expire
```

### Rule 5: Owner 才能 Renew / Release

Renew 和 Release 必须校验 Owner 身份和 Version。

### Rule 6: NodeGroup 是扩展边界

默认情况下，PlacementStrategy 只能在指定的 `NodeType + NodeGroup` 内选择节点。

跨 Group 迁移必须是显式 Transfer 或更高层策略决策。

### Rule 7: 本地缓存不是唯一真相

本地缓存只能作为加速层。

Directory 才是 Source of Truth。

当缓存和 Directory 不一致时，以 Directory 为准。

### Rule 8: 缓存必须可被主动清理

任何会改变 Placement 或节点有效性的操作，都必须能触发缓存失效。

包括：

```text
Transfer
Release
Recover
Expire
NodeDraining
NodeReplaced
NodeUnregistered
NodeMarkedInvalid
NodeRestored
```

事件订阅异常时，节点必须清空本地缓存或降级回源 Directory。

### Rule 9: 同名节点实例必须区分运行期身份

`NodeIdentity` 表示逻辑节点名。

`NodeSessionID` 表示一次具体运行实例。

当相同 `NodeIdentity` 的新 `NodeSessionID` 注册时，旧 session 相关续约必须失败，并触发旧节点信息清理。

因此 Renew / Release 这类 Owner 操作不能只校验 `NodeIdentity`，还必须校验 `NodeSessionID`。

### Rule 10: Allocate 必须过滤失效节点

根据 `NodeType + NodeGroup` 分配有效节点时，必须先过滤失效组里的 NodeName。

Strategy 不能看到已经失效的节点。

失效组按 NodeName 生效，不能因为同名节点换了新的 NodeSessionID 就自动恢复。

### Rule 11: 失效节点不影响已有 Placement 的确定性

节点进入失效组后，已有 Placement 不应被 Lookup 自动修改。

已有 Placement 只能通过显式命令迁移或释放：

```text
Transfer
Release
Recover
Expire
```

## 10. 功能需求

### 10.1 Node 能力

必须支持：

```go
RegisterNode()
RenewNode()
UnregisterNode()
ListNodes()
FindNodes(nodeType, nodeGroup)
ReplaceNodeSession()
MarkNodeInvalid(nodeType, nodeGroup, nodeName)
RestoreNode(nodeType, nodeGroup, nodeName)
ListInvalidNodes(nodeType, nodeGroup)
```

### 10.2 Placement 能力

必须支持：

```go
Lookup()
Allocate()
Renew(nodeIdentity, nodeSessionID, placementVersion, leaseVersion)
Release(nodeIdentity, nodeSessionID, placementVersion, leaseVersion)
Transfer()
Recover()
Exists()
FindByNode(nodeIdentity, statusFilter, cursor, limit)
FindByKind()
FindByGroup()
```

`FindByNode(nodeIdentity, statusFilter, cursor, limit)` 必须按完整 NodeIdentity 返回该节点承载的 Placement，供缩容迁移按节点逐步 Transfer / Release / Recover 使用。

第一阶段至少需要支持：

- 按 NodeIdentity 查询 Active Placement。
- 通过 `cursor + limit` 分页遍历。
- 查询过程不退化为全量扫描。

### 10.3 Strategy 能力

第一阶段至少支持：

```text
RoundRobin
Hash
Weight
```

后续可扩展：

```text
LeastLoad
Affinity
Zone
Rebalance
```

### 10.4 Local Cache 能力

节点侧需要支持本地 Placement 缓存。

必须支持：

```go
GetCachedPlacement(grainKey)
SetCachedPlacement(grainKey, placement)
DeleteCachedPlacement(grainKey)
DeleteCachedPlacementsByNode(nodeIdentity)
DeleteCachedPlacementsByGroup(nodeType, nodeGroup)
ClearPlacementCache()
```

缓存写入时机：

- Allocate 成功后。
- Lookup 回源 Directory 成功后。
- 仅在事件订阅健康、节点未处于降级回源模式时允许写入。

缓存删除时机：

- 收到 Placement 变更事件。
- 收到 NodeDraining 事件。
- 收到 NodeReplaced 事件。
- 收到 NodeMarkedInvalid 事件。
- 收到 NodeRestored 事件。
- 事件订阅异常或无法确认事件连续性。
- 本地判断 Lease 过期。
- 人工主动清理。

### 10.5 Event Subscribe 能力

节点需要能够订阅失效事件。

第一阶段可以使用 Redis Pub/Sub。

节点至少需要处理：

```text
PlacementTransferred
PlacementReleased
PlacementRecovered
LeaseExpired
NodeDraining
NodeReplaced
NodeUnregistered
NodeMarkedInvalid
NodeRestored
ManualCacheClear
```

订阅异常处理要求：

- 订阅异常时必须清空本地缓存或进入降级回源模式。
- 降级回源期间 Lookup 必须直接查询 Directory。
- 降级回源期间禁止写入本地缓存。
- 重新启用缓存前必须清空本地缓存，并确认订阅已经恢复健康。

## 11. 非功能需求

### 11.1 可扩展性

需要支持十万在线规模。

设计上要求：

- 支持节点横向增加。
- Lookup 延迟稳定。
- Allocate 不产生全局锁竞争。
- 数据结构避免全量扫描。
- FindByNode 必须能通过 NodeIdentity 索引或等价结构查询，不能扫描所有 Placement。

### 11.2 可恢复性

节点故障后：

- Lease 会过期。
- Placement 可被 Recover。
- 新 Owner 需要基于 Version 创建或接管 Placement。

### 11.3 可观测性

需要产生领域事件：

```text
NodeRegistered
NodeReplaced
NodeDraining
NodeMarkedInvalid
NodeRestored
NodeUnregistered
PlacementCreated
PlacementRenewed
PlacementReleased
PlacementTransferred
PlacementRecovered
LeaseExpired
PlacementCacheInvalidated
ManualCacheClear
```

这些事件可用于：

- Dashboard。
- Debug。
- 审计。
- 运维排障。

### 11.4 存储可替换

底层存储不是领域模型的一部分。

第一阶段可以实现：

```text
MemoryDirectory
```

后续实现：

```text
RedisDirectory
EtcdDirectory
DatabaseDirectory
```

## 12. 第一阶段范围

第一阶段建议只做最小闭环：

- 定义领域模型。
- 定义 Directory 接口。
- 定义 NodeIdentity。
- 定义 NodeSessionID。
- 定义 Placement 状态。
- 定义本地 Placement 缓存接口。
- 定义缓存失效事件。
- 定义节点失效组。
- 实现 MemoryDirectory。
- 实现 MemoryPlacementCache。
- 实现一个简单 Strategy。
- 验证 Lookup / Allocate / Renew / Release / Transfer。
- 验证 Allocate 后写入本地缓存。
- 验证缓存失效事件可以清理指定 Grain 或指定 NodeIdentity。
- 验证失效组会过滤 Allocate 候选节点。
- 验证 FindByNode 可以按 NodeIdentity 分页查询 Active Placement。
- 不绑定 protoactor-go。
- 提供 protoactor-go adapter 设计预留。

暂不做：

- 自动 Rebalance。
- Dashboard。
- 多存储实现。
- 跨机压测。
- 复杂负载算法。

## 13. 验收标准

第一阶段完成时，需要满足：

- 同一个 Grain 重复 Allocate 只能得到同一个 Active Placement。
- Lookup 不会改变 Placement。
- Release 后可以重新 Allocate。
- Transfer 后 Lookup 返回新 Owner。
- 旧 Owner 无法 Renew 新 Version。
- NodeGroup 内可以横向增加节点。
- Strategy 只在指定 NodeType + NodeGroup 内选择节点。
- protoactor-go 与非 protoactor-go 场景都能通过接口表达。
- Allocate 成功后，本地缓存可以命中该 Grain 的 Placement。
- Placement 发生 Transfer / Release / Recover 后，本地缓存会被失效。
- NodeDraining 可以按 NodeIdentity 主动清理缓存。
- 同名节点以新 NodeSessionID 启动后，旧 session 续约失败，旧节点相关缓存被清理。
- MarkNodeInvalid 后，该 NodeName 不再参与新的 Allocate。
- MarkNodeInvalid 对 NodeName 生效，节点换新的 NodeSessionID 后仍然无效。
- MarkNodeInvalid 不会让已有 Placement 在 Lookup 时自动迁移。
- MarkNodeInvalid 必须发布 NodeMarkedInvalid 事件。
- RestoreNode 后，该 NodeName 可以重新参与新的 Allocate。
- RestoreNode 必须发布 NodeRestored 事件。
- 某个 NodeType + NodeGroup 下所有节点都失效时，Allocate 返回无可用节点错误。
- 事件订阅异常后，本地缓存会被清空或 Lookup 降级回源 Directory。
- 缩容迁移可以通过 FindByNode(NodeIdentity) 找到目标节点承载的 Placement。
- FindByNode(NodeIdentity) 默认返回 Active Placement，并支持分页遍历。

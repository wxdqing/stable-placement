# Stable Placement 开发需求文档

本文档面向开发落地。

当前版本使用 Node Lease v2：Lease 只属于 NodeSession，旧 Grain Lease、heartbeat、Expire 和 Placement Expired 语义已被取代，不是当前能力。

它不按 DDD 方式组织，而按实现者的阅读顺序组织：

```text
先知道做什么
再知道有哪些概念
再知道数据结构和接口
再知道每个流程怎么跑
最后知道怎么测试和验收
```

相关文档：

- `需求.md`：完整需求和规则来源。
- `ontology.md`：概念模型、边界和关系说明。
- `logging.md`：logger 注入、错误记录边界和数据安全约定。

---

## 1. 文档目标

第一阶段目标是实现一个最小可用的 Stable Placement 组件。

它需要支持：

- 注册节点。
- 查询 Placement。
- 首次分配 Placement。
- Owner 续约。
- Owner 释放。
- 显式迁移。
- 故障恢复。
- 节点失效组。
- 本地缓存。
- 缓存失效事件。
- 按 NodeIdentity 查询 Placement，支持缩容迁移。

第一阶段不追求：

- Redis / etcd / 数据库实现。
- 自动 Rebalance。
- Dashboard。
- 跨机压测。
- 复杂负载算法。
- protoactor-go 深度绑定。

## 4. 核心概念

### 4.1 Grain

Grain 是业务逻辑实体。

示例：

```text
Player
Guild
Mail
Rank
Scene
```

关键字段：

```go
type Grain struct {
    GrainID         string
    Kind            string
    TargetNodeType  string
    TargetNodeGroup string
}
```

说明：

- Grain 不关心自己在哪个节点。
- `Kind + GrainID` 组成 `GrainKey`。
- `TargetNodeType + TargetNodeGroup` 用于确定候选节点集合。

### 4.2 Node

Node 是运行节点。

关键字段：

```go
type Node struct {
    NodeType        string
    NodeGroup       string
    NodeName        string
    NodeIdentity    string
    NodeSessionID   string
    Address         string
    Weight          int
    Load            int
    Status          NodeStatus
    Lease           NodeLease
}

type NodeLease struct {
    Version           int64
    TTLMillis         int64
    ExpireAtUnixMilli int64
}
```

说明：

- `NodeIdentity` 是稳定逻辑节点名。
- `NodeSessionID` 是一次具体运行实例。
- 同名节点重启后，`NodeIdentity` 相同，但 `NodeSessionID` 必须不同。
- 旧 `NodeSessionID` 的 Renew / Release 必须失败。
- Node Lease 隐式属于当前 NodeSession，是唯一 TTL 所有者。

### 4.3 Placement

Placement 是 Grain 到 Node 的稳定归属关系。

关键字段：

```go
type Placement struct {
    GrainID            string
    Kind               string
    GrainKey           GrainKey
    NodeIdentity       string
    OwnerNodeSessionID string
    Version            int64
    Status             PlacementStatus
    CreateTime         time.Time
    UpdateTime         time.Time
}
```

说明：

- Placement 不是 Actor。
- Placement 是路由归属关系。
- Placement 保存完整 `NodeIdentity`。
- Placement 保存分配时的 Owner session 快照，但不保存 TTL。
- Node Lease 到期只使 Placement 逻辑不可路由，Placement 记录和 Active 状态继续保留。

### 4.4 NodeLease 与 PlacementRoute

NodeLease 保护一个 NodeSession 承载的全部 Grain。PlacementRoute 是 Lookup 返回的进程内可路由快照。

```go
type PlacementRoute struct {
    GrainKey           GrainKey
    NodeIdentity       string
    OwnerNodeSessionID string
    Version            int64
    Status             PlacementStatus
    NodeLeaseVersion   int64
    ValidUntil         time.Time
}
```

要求：

- `NodeLeaseConfig` 在实例构造时不可变，默认一分钟，`TTL <= 0` 返回 `ErrInvalidNodeLeaseTTL`。
- RegisterNode 和 ReplaceNodeSession 将 TTL 持久化为 `Node.Lease.TTLMillis`。
- RenewNode 使用持久化 TTLMillis，过期 session 返回 `ErrNodeLeaseExpired`，不能复活。
- Redis 所有 Lease 判断以 Lua 内的 Redis `TIME` 为权威；Memory 使用内部 clock。
- ValidUntil 是当前进程的保守截止时间，不持久化。

### 4.5 LocalPlacementCache

LocalPlacementCache 是节点本地缓存。

缓存内容：

```text
GrainKey -> PlacementRoute
NodeIdentity -> GrainKey Set
```

缓存不是唯一真相。

Directory 才是唯一真相。

## 5. 身份模型

### 5.1 GrainKey

```text
GrainKey = Kind + "/" + GrainID
```

示例：

```text
Player/10001
Guild/9001
```

### 5.2 NodeIdentity

```text
NodeIdentity = NodeType + "/" + NodeGroup + "/" + NodeName
```

示例：

```text
game/default/game-1
game/default/game-2
chat/world/chat-1
```

要求：

- 全局唯一。
- Placement 必须保存完整 NodeIdentity。
- 缓存可以按 NodeIdentity 清理。
- FindByNode 必须按 NodeIdentity 查询。

### 5.3 NodeSessionID

`NodeSessionID` 表示一次具体运行实例。

要求：

- 同名节点每次重新启动，都必须生成新的 NodeSessionID。
- `NodeIdentity` 相同但 `NodeSessionID` 不同，表示不是同一个运行实例。
- 旧 session 的 Renew / Release 必须失败。

## 6. 状态定义

### 6.1 NodeStatus

建议第一阶段支持：

```go
type NodeStatus string

const (
    NodeStatusActive   NodeStatus = "active"
    NodeStatusDraining NodeStatus = "draining"
    NodeStatusOffline  NodeStatus = "offline"
)
```

说明：

- `active`：可以参与 Allocate。
- `draining`：不应该参与新的 Allocate。
- `offline`：不可用。

最终是否可分配，还必须经过 InvalidNodeGroup 过滤。

### 6.2 PlacementStatus

建议第一阶段支持：

```go
type PlacementStatus string

const (
    PlacementStatusActive   PlacementStatus = "active"
    PlacementStatusReleased PlacementStatus = "released"
)
```

说明：

- `active`：当前有效归属。
- `released`：Owner 主动释放。

Active 只表示归属记录未释放；是否可路由还必须校验 Owner Node、session、状态和 Node Lease。

## 7. 核心接口

### 7.1 Directory

Directory 是唯一真相。

建议接口：

```go
type Directory interface {
    Lookup(ctx context.Context, key GrainKey) (*PlacementRoute, error)
    Allocate(ctx context.Context, cmd AllocateCommand) (*Placement, error)
    Renew(ctx context.Context, cmd RenewCommand) (*Placement, error)
    Release(ctx context.Context, cmd ReleaseCommand) error
    Transfer(ctx context.Context, cmd TransferCommand) (*Placement, error)
    Recover(ctx context.Context, cmd RecoverCommand) (*Placement, error)
    Exists(ctx context.Context, key GrainKey) (bool, error)
    FindByNode(ctx context.Context, query FindByNodeQuery) (PlacementPage, error)
}
```

第一阶段可以只实现 MemoryDirectory。

### 7.2 NodeRegistry

```go
type NodeRegistry interface {
    RegisterNode(ctx context.Context, node Node) error
    RenewNode(ctx context.Context, nodeIdentity string, nodeSessionID string) error
    ExpireNodeLeases(ctx context.Context, nodeType string, nodeGroup string, limit int64) (int, error)
    UnregisterNode(ctx context.Context, nodeIdentity string, nodeSessionID string) error
    ReplaceNodeSession(ctx context.Context, node Node) (*Node, error)
    FindNodes(ctx context.Context, nodeType string, nodeGroup string) ([]Node, error)
    DrainNode(ctx context.Context, nodeIdentity string) error
    CompleteDrain(ctx context.Context, nodeIdentity string, nodeSessionID string) error
    MarkNodeInvalid(ctx context.Context, nodeType string, nodeGroup string, nodeName string) error
    RestoreNode(ctx context.Context, nodeType string, nodeGroup string, nodeName string) error
    ListInvalidNodes(ctx context.Context, nodeType string, nodeGroup string) ([]string, error)
}
```

要求：

- RegisterNode 只表示节点在线。
- 是否可参与 Allocate，要继续检查 NodeStatus 和 InvalidNodeGroup。
- InvalidNodeGroup 按 NodeName 生效，跨 NodeSessionID 持续有效。

### 7.3 PlacementCache

```go
type PlacementCache interface {
    GetCachedPlacement(key GrainKey) (*PlacementRoute, bool)
    SetCachedPlacement(key GrainKey, route PlacementRoute)
    DeleteCachedPlacement(key GrainKey)
    DeleteCachedPlacementsByNode(nodeIdentity string)
    DeleteCachedPlacementsByGroup(nodeType string, nodeGroup string)
    ClearPlacementCache()
    SetDegraded(degraded bool)
    IsDegraded() bool
}
```

要求：

- 降级回源模式下，不能读缓存。
- 降级回源模式下，不能写缓存。
- 订阅恢复后，必须先 Clear，再退出 degraded。

### 7.4 EventBus

```go
type EventPublisher interface {
    Publish(ctx context.Context, event PlacementEvent) error
}

type EventSubscriber interface {
    Subscribe(ctx context.Context, handler func(PlacementEvent) error) error
}
```

第一阶段可以实现 MemoryEventBus。

订阅异常时，调用方必须清空缓存或进入降级回源模式。

### 7.5 PlacementStrategy

```go
type PlacementStrategy interface {
    Choose(ctx context.Context, input StrategyInput) (Node, error)
}
```

要求：

- Strategy 只看到过滤后的 EffectiveNodes。
- Strategy 不参与 Lookup。
- 第一阶段实现 RoundRobin 即可。

## 9. 核心流程

### 9.1 RegisterNode

流程：

```text
Node starts
  -> RegisterNode
  -> validate NodeIdentity metadata
  -> Save NodeIdentity + NodeSessionID + NodeLease(TTLMillis)
  -> Node becomes Active
```

注意：

- 注册成功不等于可分配。
- Allocate 时仍要过滤 InvalidNodeGroup。
- 同 session 重试幂等但不续约；不同 session 必须使用 ReplaceNodeSession。
- Offline 或已过期的相同 session 不能通过 RegisterNode 复活。

### 9.2 ReplaceNodeSession

流程：

```text
same NodeIdentity new NodeSessionID
  -> Replace old session
  -> Publish NodeReplaced
  -> old session Renew / Release fails
  -> cache clears by NodeIdentity
```

要求：

- 新 session 不能继承旧 session 的运行期权限。
- 新 session 创建独立的 Version=1 Node Lease。
- 旧 Placement 的 OwnerNodeSessionID 不变，因此新 session 不继承旧 Placement。
- 旧 session 后续 Renew 必须失败。

### 9.3 Lookup

流程：

```text
Lookup(GrainKey)
  -> if cache healthy, try LocalPlacementCache
  -> if cache miss or degraded, query Directory
  -> if cache healthy, write cache
```

要求：

- Lookup 无副作用。
- Lookup 不创建 Placement。
- Lookup 不迁移 Placement。
- Lookup 不恢复 Placement。
- 降级回源模式下不能读写缓存。
- 返回 PlacementRoute；缓存或调用方只可在当前时间早于 ValidUntil 时使用。
- Redis 用同一次 Redis `TIME` 计算 remaining TTL，并从请求开始时间换算 ValidUntil；网络耗时只缩短窗口。
- 逻辑失效 Placement 返回 `ErrPlacementNotFound`，但不删除或改写记录。

### 9.4 Allocate

流程：

```text
Allocate(Grain)
  -> check existing active Placement
  -> FindNodes(NodeType, NodeGroup)
  -> remove inactive nodes
  -> remove InvalidNodeGroup NodeName
  -> Strategy Choose
  -> Create Placement
  -> Publish PlacementCreated
  -> write cache if cache healthy
```

要求：

- 已存在 Active Placement 时，不能创建第二个。
- 已存在 Active 但 Owner Node/session/Lease 不可用时返回 `ErrPlacementOwnerUnavailable`，不能自动换 Owner。
- EffectiveNodes 为空时返回 `ErrNoAvailableNode`。
- Strategy 只能看到 EffectiveNodes。

### 9.5 Renew

流程：

```text
Renew
  -> Lookup Placement
  -> check NodeIdentity
  -> check NodeSessionID
  -> check PlacementVersion
  -> check current Node Lease is valid
  -> keep Placement and TTL unchanged
  -> publish PlacementRenewed
```

要求：

- 旧 Owner 不能续约新 Version。
- 旧 NodeSessionID 不能续约新 session。
- Directory.Renew 只校验并审计；Node TTL 只能由 RenewNode 续约。

### 9.6 Release

流程：

```text
Release
  -> check Owner identity and session
  -> check Version
  -> mark released
  -> publish PlacementReleased
  -> invalidate cache by GrainKey
```

Owner Node 仍存在且 session 未替换时，即使 Node Offline 或 Lease 到期也允许 Release；Node 已删除时使用 Transfer 或 Recover。

### 9.7 Transfer

流程：

```text
Transfer
  -> check PlacementVersion
  -> check target NodeIdentity is effective
  -> update NodeIdentity
  -> bump Version
  -> publish PlacementTransferred
  -> invalidate cache by GrainKey
```

要求：

- Transfer 必须是显式命令。
- Lookup 不能触发 Transfer。
- 缩容迁移时，先 FindByNode，再逐个 Transfer。

### 9.8 Recover

流程：

```text
Recover
  -> old Owner unreliable
  -> check Version and Owner is logically unavailable
  -> set new Owner
  -> bump Version
  -> publish PlacementRecovered
  -> invalidate cache by GrainKey
```

Recover 只用于 missing、Offline、Lease 到期或 session 不匹配的 Owner；健康 Owner 必须使用显式 Transfer。Transfer 可迁移健康或不可用 Owner。

### 9.9 MarkNodeInvalid

流程：

```text
MarkNodeInvalid(NodeType, NodeGroup, NodeName)
  -> add NodeName to InvalidNodeGroup
  -> publish NodeMarkedInvalid
  -> invalidate cache by NodeIdentity
```

要求：

- 只影响新的 Allocate。
- 不自动迁移已有 Placement。
- 失效状态按 NodeName 生效，跨 NodeSessionID 持续有效。

### 9.10 RestoreNode

流程：

```text
RestoreNode(NodeType, NodeGroup, NodeName)
  -> remove NodeName from InvalidNodeGroup
  -> publish NodeRestored
```

要求：

- Restore 后，该 NodeName 可以重新参与 Allocate。
- 必须发布 NodeRestored。

## 10. 缓存机制

### 10.1 缓存写入

允许写入缓存的场景：

- Allocate 成功后。
- Lookup 回源 Directory 成功后。

前提：

- 事件订阅健康。
- 本地缓存未处于 degraded 模式。

### 10.2 缓存清理

必须支持：

```text
Delete by GrainKey
Delete by NodeIdentity
Delete by NodeType + NodeGroup
Clear all
```

### 10.3 订阅异常处理

订阅异常包括：

- 订阅连接断开。
- 事件无法反序列化。
- 事件版本无法判断。
- 无法确认是否漏事件。
- 事件处理连续失败。

处理要求：

```text
ClearPlacementCache
or
SetDegraded(true)
```

degraded 模式下：

- Lookup 必须直接查 Directory。
- 不能读缓存。
- 不能写缓存。

恢复时：

```text
Subscribe healthy
  -> ClearPlacementCache
  -> SetDegraded(false)
```

## 13. 错误定义

建议第一阶段定义：

```go
var (
    ErrPlacementNotFound  = errors.New("placement not found")
    ErrPlacementExists    = errors.New("placement already exists")
    ErrNoAvailableNode    = errors.New("no available node")
    ErrInvalidOwner       = errors.New("invalid owner")
    ErrInvalidNodeSession = errors.New("invalid node session")
    ErrVersionConflict    = errors.New("version conflict")
    ErrNodeLeaseExpired   = errors.New("node lease expired")
    ErrInvalidNodeLeaseTTL = errors.New("invalid node lease TTL")
    ErrPlacementOwnerUnavailable = errors.New("placement owner unavailable")
    ErrNodeNotFound       = errors.New("node not found")
)
```

错误使用要求：

- 没有 Active Placement：`ErrPlacementNotFound`。
- 所有候选节点都不可用：`ErrNoAvailableNode`。
- NodeIdentity 不匹配：`ErrInvalidOwner`。
- NodeSessionID 不匹配：`ErrInvalidNodeSession`。
- PlacementVersion 不匹配：`ErrVersionConflict`。
- Node Lease 已过期：`ErrNodeLeaseExpired`。
- Active Placement 的 Owner 不可用：`ErrPlacementOwnerUnavailable`。

## 14. 测试要求

### 14.1 Lookup

- 未分配时返回 NotFound。
- 已分配时返回 Active Placement。
- Lookup 不创建 Placement。
- Lookup 不触发 Transfer / Recover。

### 14.2 Allocate

- 第一次 Allocate 创建 Placement。
- 重复 Allocate 同一个 Grain 返回已有 Active Placement 或冲突保护结果。
- EffectiveNodes 为空时返回 NoAvailableNode。
- InvalidNodeGroup 中的 NodeName 不会被选中。

### 14.3 Renew / Release

- 当前 Owner 可以 Renew。
- 旧 Version 不能 Renew。
- 旧 NodeSessionID 不能 Renew。
- Renew 只校验并审计，不修改 Placement 或 Node Lease。
- Release 必须校验 NodeIdentity + NodeSessionID。
- Release 后可以重新 Allocate。

### 14.4 Transfer / Recover

- Transfer 后 Lookup 返回新 Owner。
- Transfer 必须推进 Version。
- 旧 Owner 无法 Renew 新 Version。
- Recover 可以在旧 Owner 不可信时接管。
- Recover 拒绝健康 Owner；健康 Owner 使用 Transfer。

### 14.5 Cache

- Allocate 成功后可以写缓存。
- Lookup 回源成功后可以写缓存。
- PlacementTransferred 清理 GrainKey 缓存。
- NodeDraining 清理 NodeIdentity 缓存。
- 订阅异常后清空缓存或进入 degraded。
- degraded 下 Lookup 直接查 Directory，且不写缓存。

### 14.6 InvalidNodeGroup

- MarkNodeInvalid 后，该 NodeName 不参与新的 Allocate。
- 同名节点换新的 NodeSessionID 后仍然无效。
- RestoreNode 后，该 NodeName 可以重新参与 Allocate。
- MarkNodeInvalid / RestoreNode 必须发布事件。

### 14.7 FindByNode

- 可以按 NodeIdentity 查询 Active Placement。
- 支持分页。
- 缩容迁移可以基于 FindByNode 逐个 Transfer。

---

## 15. 验收标准

第一阶段完成时，需要满足：

- 同一个 Grain 同一时间最多只有一个 Active Placement。
- Lookup 无副作用。
- Allocate 只在没有 Active Placement 时创建。
- Renew / Release 校验 NodeIdentity + NodeSessionID + Version。
- Transfer / Recover 推进 Version。
- InvalidNodeGroup 按 NodeName 生效，跨 NodeSessionID 持续有效。
- MarkNodeInvalid / RestoreNode 发布事件。
- 本地缓存能按 GrainKey / NodeIdentity 清理。
- 事件订阅异常后，缓存清空或降级回源。
- FindByNode 可以按 NodeIdentity 分页查询 Active Placement。
- MemoryDirectory 查询不依赖全量扫描。
- Node Lease 是唯一 TTL 所有者，Memory 与 Redis 的配置、错误和到期边界一致。
- Node Lease 到期后 Placement 保留，Owner 只能通过显式 Recover、Transfer、Release 或 Released 后 Allocate 改变。

## 16. 后续扩展

Node Lease v2 使用独立 Redis namespace，并保持 Memory/Redis 契约一致。v1/v2 部署必须执行 [冷切换手册](./node-lease-v2-cutover.md)：v1 workload 未全部停止不得开放 v2，预置 v1 key 不影响 v2，本库只暴露 namespace/version 常量而不探测其他进程；`首笔 v2 业务写入` 后禁止直接回滚，该术语以手册定义为准。

后续可以扩展：

- RedisDirectory。
- Redis Pub/Sub EventBus。
- etcd Directory。
- etcd watch EventBus。
- protoactor-go adapter。
- Dashboard。
- 自动 Rebalance。
- 更复杂的 PlacementStrategy。

扩展原则：

- 先保持 MemoryDirectory 的语义正确。
- 再替换底层存储。
- 不为了未来存储提前污染核心模型。
- 不为了 adapter 提前绑定 protoactor-go。

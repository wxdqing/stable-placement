# Stable Placement Node Lease v2 开发需求文档

本文档描述当前实现契约。v1 的 Grain Lease、heartbeat、Expire 和 Placement Expired 语义已经被 Node Lease v2 取代，不是当前能力，也不提供兼容入口。

相关文档：

- `ontology.md`：领域概念和关系。
- `开发清单.md`：实现与部署验收项。
- `node-lease-v2-cutover.md`：v1 到 v2 的唯一受支持部署方式。

## 1. 目标与边界

Stable Placement 维护 Grain 到 Node 的稳定归属，并保证只有 Node Lease 有效的当前 Owner session 可以路由。

组件负责 Lookup、Allocate、Renew、Release、Transfer、Recover、Node 注册与状态、失效节点组、缓存和事件。组件不负责启动 Grain、投递业务消息、自动 Rebalance、在线迁移 v1 数据、自动恢复每个 Grain 或发现其他进程。

## 2. 核心模型

### 2.1 Node 与 Node Lease

```go
type NodeLease struct {
    Version           int64
    TTLMillis         int64
    ExpireAtUnixMilli int64
}

type Node struct {
    NodeType      string
    NodeGroup     string
    NodeName      string
    NodeIdentity  string
    NodeSessionID string
    Address       string
    Weight        int
    Load          int
    Status        NodeStatus
    Lease         NodeLease
}
```

NodeIdentity 是稳定身份，格式为 `NodeType/NodeGroup/NodeName`；NodeSessionID 标识一次运行实例。Node Lease 隐式属于该 session，不属于任何 Grain。

### 2.2 Placement 与路由

```go
type Placement struct {
    GrainID            string
    Kind               string
    GrainKey           GrainKey
    NodeIdentity       string
    OwnerNodeSessionID string
    Version            int64
    Status             PlacementStatus
    CreateTime         time.Time
    UpdateTime         time.Time
}

type PlacementRoute struct {
    GrainKey           GrainKey
    NodeIdentity       string
    OwnerNodeSessionID string
    Version            int64
    Status             PlacementStatus
    NodeLeaseVersion   int64
    ValidUntil         time.Time
}
```

Placement 只保存归属和 Owner session 快照，不保存 TTL。状态只有 Active 和 Released。Node Lease 到期不会改写 Placement；记录继续保留，供显式 Recover、Transfer 和运维查询使用。

PlacementRoute 是当前进程的可路由授权。Allocate、Renew、Transfer 和 Recover 返回的 Placement 不能直接用作路由，调用方必须再 Lookup。

## 3. TTL 与权威时间

`NodeLeaseConfig` 只在构造 Memory NodeRegistry 或 Redis Directory 时传入，构造后不可变。`DefaultNodeLeaseConfig()` 返回一分钟；`TTL <= 0` 必须返回 `ErrInvalidNodeLeaseTTL`。

RegisterNode 和 ReplaceNodeSession 将构造配置换算为毫秒，持久化到 `Node.Lease.TTLMillis`。RenewNode 必须使用该持久值续约，不能使用执行续约的进程配置，因此不同配置的进程不会改变既有 session 的 TTL 语义。

Redis 的 RegisterNode、RenewNode、Lookup、Allocate、Transfer、Recover 和 ExpireNodeLeases 都以 Lua 内的 Redis `TIME` 为权威时间，不接受应用传入的当前时间。Memory 使用构造时注入的内部 clock，并在一次操作内使用同一快照。两种实现都在到期边界立即判定失效。

RenewNode 只能续约尚未过期的 Active 或 Draining session。已过期、Offline 或 session 不匹配时必须失败；特别是已过期 session 返回 `ErrNodeLeaseExpired`，不能被续约复活。

## 4. Node 生命周期

### RegisterNode

- 严格校验 NodeIdentity 元数据。
- 不存在的 identity 创建 Active Node，Lease 版本为 1。
- 相同 session 重试幂等成功，但不延长 Lease、不重复发布 NodeRegistered。
- 不同 session 返回 `ErrInvalidNodeSession`；必须显式调用 ReplaceNodeSession。
- Offline 或 Lease 已过期的相同 session 返回 `ErrNodeLeaseExpired`。

### ReplaceNodeSession

- old 与 new session 必须不同。
- 新 session 创建版本为 1 的独立 Node Lease并进入 Active。
- InvalidNodeGroup 按 NodeName 保留。
- 发布 NodeReplaced 并清除该 NodeIdentity 的缓存。
- 旧 Placement 的 OwnerNodeSessionID 不改变，因此新 session 不继承旧 Placement。

### ExpireNodeLeases

扫描器只负责把到期的 Active/Draining Node 原子推进为 Offline 并发布一次 NodeLeaseExpired。逻辑失效不依赖扫描器：任何路由和 Owner 校验都必须直接检查 Lease 到期时间。

Offline Node JSON 和 Placement 索引保留。应用负责按需启动 `RunNodeLeaseLoop`；库不自动启动后台循环。

## 5. Placement 语义

Placement 可路由必须同时满足：状态 Active；Owner Node 存在；OwnerNodeSessionID 与当前 NodeSessionID 相等；Node 为 Active 或 Draining；权威时间早于 Node Lease 到期时间。

- Lookup 无副作用，逻辑失效时返回 `ErrPlacementNotFound`。
- Exists 对逻辑失效 Placement 返回 false。
- Allocate 只选择 Active、未被标记失效且 Lease 有效的 Node。已有 Active Placement 的 Owner 不可用时返回 `ErrPlacementOwnerUnavailable`，不自动换 Owner。
- Renew 只校验 Placement、Owner、session、版本和 Node Lease，并产生 PlacementRenewed 审计；不修改 Placement 或 TTL。
- Release 校验当前 session 和 PlacementVersion。Node 仍存在且 session 未替换时，即使 Offline 或 Lease 到期也允许释放。
- Transfer 是显式管理迁移，可迁移健康或不可用 Owner；目标必须可分配。
- Recover 只允许 Active 且 Owner 缺失、Offline、Lease 到期或 session 不匹配的 Placement。健康 Owner 返回 `ErrPlacementNotRecoverable`，应使用 Transfer。

任何 Owner 变化都必须由 Transfer、Recover、Release 或 Released 后重新 Allocate 显式发生。

## 6. 缓存与事件

Redis Lookup 在一个只读 Lua 快照中用 Redis `TIME` 计算剩余 TTL。Go 在请求开始前记录带 monotonic 信息的时间，并以它加剩余 TTL 生成 `ValidUntil`；网络往返只会缩短可用窗口。响应到达时如果窗口已结束，Lookup 返回 `ErrPlacementNotFound`，不写缓存。

Memory Lookup 使用同一个内部 clock 快照判断 Lease 并生成 ValidUntil。所有调用方和 CachedRouter 在使用 route 前都必须检查当前时间早于 ValidUntil。

NodeLeaseExpired、NodeReplaced、NodeDraining、NodeMarkedInvalid 和 NodeUnregistered 按 NodeIdentity 清缓存；NodeRestored 按 NodeType + NodeGroup 清缓存；未知或字段不完整事件清空全部缓存。PlacementRenewed 只用于审计，CachedRouter 忽略它。

## 7. Memory / Redis 一致性

Memory 和 Redis 必须对公共 API、错误、TTL、NodeIdentity 校验、session 规则、到期边界、路由资格、Recover/Transfer 行为保持一致。实现机制可以不同：Memory 使用固定锁顺序和内部 clock，Redis 使用同 slot Lua、Redis `TIME`、v2 Stream 和 outbox。

## 8. Redis v2 namespace

Redis 数据使用 `sp:{stable-placement}:v2:` 前缀。v2 不读取、修改或删除 v1 key，consumer group 和 Pub/Sub channel 也使用 v2 名称。

库只暴露 `redis.NamespaceVersion` 和 `redis.NamespacePrefix` 常量供部署系统检查。库不扫描 v1 namespace，不探测 v1 进程，也不决定是否开放流量。部署门禁和 v1 数据的最终人工清理由运维系统负责。

## 9. 部署与验收

v1/v2 不能滚动升级或同时接受业务流量。部署必须执行 [Node Lease v2 冷切换手册](./node-lease-v2-cutover.md)。关键门禁是：v1 writer、Node、scanner、consumer 未全部停止时不得开放 v2；预置 v1 key 不得影响 v2；全部 v2 writer、Node、scanner 和 consumer 必须整体启动并健康；首笔 v2 业务写入后不得直接回滚到 v1。

完整代码验收包括 Memory/Redis 契约、真实 Redis、race 和 vet。真实 Redis 默认地址为 `127.0.0.1:6379`，密码只能从 `STABLE_PLACEMENT_REDIS_PASSWORD` 或 `STABLE_PLACEMENT_REAL_REDIS_PASSWORD` 环境变量注入，禁止写入文档、命令历史或仓库配置。

## 10. v1 历史语义

旧文档中的 Grain Lease、LeaseVersion、LeaseTTL、ExtendTTL、heartbeat、Expire、ExpireDue、ExpireHeartbeats、ErrLeaseExpired 和 PlacementStatusExpired 全部是 v1 历史概念。v2 不保留这些字段、API 或兼容行为。
